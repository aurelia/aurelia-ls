# Authoring Capability Checklist

This is a living scope map for AI-assisted Aurelia app authoring. It is not an implementation order and not a promise
that every item belongs in one API surface. Update it when capability boundaries move, and promote stable rules to
[README.md](./README.md), [ONTOLOGY.md](./ONTOLOGY.md), or source contracts.

## Core Loop

- [ ] Capability query: report supported, partial, and open authoring operations from typed descriptors.
- [ ] Profile negotiation: expose decorator, convention, explicit-registration, and hybrid choices with ambiguity
  points.
- [ ] Recipe model: compose operations into named flows without turning large requests into magic primitives.
- [ ] Plan builder: produce semantic operations, preconditions, expected effects, and topology expectations before file
  edits.
- [ ] Source edit boundary: define how plans become concrete file edits without hiding formatting, package-manager, or
  user-taste policy inside semantic records.
- [ ] Reopen loop: apply edits externally, reopen through `api`, compare observed facts with expected effects.
- [ ] Repair loop: turn verification failures and open seams into follow-up operations.

## App Shell And Tooling

- [ ] Package manifest, dependency, script, and package-manager authoring.
- [ ] Build-tool profile selection and asset pipeline expectations.
- [ ] TypeScript config, path aliases, module resolution, and `.html` or `.css` module declaration support.
- [ ] Folder structure and naming convention policy.
- [ ] Entrypoint creation through real Aurelia imports and runtime-shaped startup.
- [ ] Root component source, external template, optional stylesheet, and host element.
- [ ] Environment/configuration file boundaries.

## Application Topology

- [ ] App files, imports, entrypoints, root components, external templates, styles, services, registrations, routes, and
  assets are represented as topology nodes.
- [ ] Existing app analysis can recover topology from source instead of only consuming topology generated by authoring.
- [ ] Topology nodes can map to source addresses for navigation, rename, and repair.
- [ ] Topology deltas can express create, connect, configure, modify, remove, verify, repair, and migrate operations.

## Resource Authoring

- [ ] Custom elements with external templates, bindables, dependencies, lifecycle, and resource visibility.
- [ ] Custom attributes with bindables, default property, usage examples, and DI dependencies.
- [ ] Template controllers with synthetic-view implications and TypeChecker handoff pressure.
- [ ] Value converters with expression typing and template usage verification.
- [ ] Binding behaviors with binding-pipeline implications and usage verification.
- [ ] Binding commands and attribute patterns only when the app intentionally needs syntax-level extension.
- [ ] Conventions, decorators, and explicit registration are modeled as profile choices, not competing hidden paths.

## Template Composition

- [ ] Binding expressions, binding commands, events, refs, interpolation, and value converters.
- [ ] Control flow and structural resources such as repeat/if/switch-like patterns and template-controller scopes.
- [ ] Forms, inputs, two-way binding, validation, submit flows, loading, empty, and error states.
- [ ] Component composition, content projection or slot-like patterns, and nested resource visibility.
- [ ] Template-to-TypeChecker handoff for member lookup, autocomplete, diagnostics, and speculative synthetic views.
- [ ] Accessibility-relevant template obligations such as labels, roles, names, focus, and keyboard paths.

## Domain, Dependency Injection, And Integration

- [ ] Domain model classes, state classes, repositories, use-case classes, and app services.
- [ ] DI keys, registrations, lifetimes, aliases, factories, injection sites, and container visibility.
- [ ] Service-to-component wiring with constructor injection and expected dependency reads.
- [ ] Backend or platform integration boundaries, including client adapters and environment-dependent configuration.
- [ ] App state strategy that starts with vanilla classes and admits framework packages only when the operation requires
  them.

## Routing, Auth, And Access Control

- [ ] Router admission and route tree authoring.
- [ ] Routeable components, layouts, navigation links, route params, and viewport structure.
- [ ] Route hooks and lifecycle/validation behavior independent of auth.
- [ ] Authentication services, provider integration, session/user state, login/logout UI, and persistence policy.
- [ ] Access-control policy, protected route admission, guard hooks, permission checks, denied-state UI, and repair
  guidance.
- [ ] Explicit ambiguity handling for server-side auth contract, token/cookie policy, and redirect behavior.

## Design And UI Policy

- [ ] Authoring profiles for visual density, layout system, styling strategy, and component-library use.
- [ ] Theme tokens, responsive behavior, dark/light/system mode, and durable CSS ownership.
- [ ] Accessibility defaults and verification expectations.
- [ ] Design choices stay in profile/policy space unless they create framework or semantic obligations.

## Evolution Operations

- [ ] Rename and move resources while preserving imports, registrations, templates, and route references.
- [ ] Extract component from an existing template and verify binding/resource preservation.
- [ ] Split inline templates into external templates and prove semantic equivalence.
- [ ] Convert between decorator, convention, and explicit registration styles when safe.
- [ ] Add routing around an existing component.
- [ ] Upgrade package/configuration shapes with app reopen verification.

## Analysis Prerequisites

- [ ] Package-aware import resolution and local source/module asset resolution.
- [ ] External template and stylesheet recognition with source-address and provenance records.
- [ ] Convention recognition for components, templates, styles, and resources.
- [ ] Package/plugin admission recognition for router, i18n, validation, state, fetch-client, and other common Aurelia
  packages.
- [ ] DI, compiler-world, resource visibility, template compilation, rendering, and TypeChecker handoff answers are
  compact enough for AI callers.
- [ ] Open seams are actionable enough to drive repair operations rather than only diagnostics.

## Fixtures And Pressure

- [ ] Authoring fixtures contain idiomatic apps we would recommend to users.
- [ ] Stress fixtures are separate and may be dense or artificial.
- [ ] Fixtures are used to pressure semantic closure and repair, not brittle snapshots.
- [ ] Each fixture documents which capability families it exercises.
