# Proof Basis

Use this note when deciding what evidence a framework-aware answer must carry
before it is allowed to close positively, negatively, or as an honest open
front.

## Why This Exists

Before scanner or evaluator healing is allowed to claim better framework-aware
answers, the package needs a local proof basis for what that healing must
actually prove.

That proof basis is not merely "support more patterns." It is about keeping
different kinds of closure separate.

## Required Proof Families

At minimum the package should distinguish:

1. positive owner-truth proof for exact framework surfaces
2. negative lookalike proof for wrong, unresolved, type-shape-only, and
   name-shape-only cases
3. same-package disambiguation proof for cases where provenance closes to one
   real framework surface but does not yet license the requested meaning
4. open, unsupported, and runtime-only proof for families whose honest
   behavior stops short of closed semantic population

## Grounding And Admissibility Stay Separate

The package must keep these distinct:

- grounded owner-surface identity
- same-package but wrong export or wrong face
- grounded owner surface with invalid member or slot
- later corridor-local semantic consequence

Examples of admissibility sentinels include:

- `Registration.singleton` versus `DI.singleton`
- `IContainer` versus `Container`
- `AppTask` versus `IAppTask`
- valid `AppTask` slot names versus invented ones

These are not all provenance failures. Some prove that provenance alone does
not close the later claim.

## Expected Local Outcome Shapes

Framework-aware results in this package should be able to represent at least:

- grounded-and-spendable
- grounded-but-inadmissible
- wrong-surface
- wrong-member-or-slot
- unresolved
- type-shape-only
- name-shape-only
- open
- unsupported
- runtime-only

The exact wire shapes can differ. The proof distinction should remain visible.

## Testing Consequence

As framework-aware surfaces land, tests should pin both:

- positive exact-grounding fixtures
- negative lookalike fixtures

That includes same-package and same-family near misses, not just obvious
"not found" cases.

If a future surface can only show positive examples and vague misses, the proof
basis is still too weak.
