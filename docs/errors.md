# Aurelia Error Code Inventory

This file inventories the AUR* error codes defined in the framework as of this snapshot.
Each entry lists the numeric code, its identifier, and the normative condition that triggers it so the LSP/typechecker can reason about throws via static analysis. Where a code is only detectable at runtime (e.g., depends on user data, environment, or callbacks), that is called out explicitly.

## Dialog (`packages/dialog/src/errors.ts`)
- `AUR0099` (`method_not_implemented`): Invoked stub method that has not been implemented/overridden.
- `AUR0901` (`dialog_not_all_dialogs_closed`): Application deactivation ran while one or more dialog controllers remained open.
- `AUR0903` (`dialog_settings_invalid`): Dialog was opened without providing `component` or `template` (at least one required).
- `AUR0904` (`dialog_no_empty_default_configuration`): Default dialog configuration was requested without specifying a renderer and without using the standard/classic configuration helpers.
- `AUR0905` (`dialog_activation_rejected`): `canActivate`/activation promise for the dialog rejected or returned a falsy veto (runtime-only).
- `AUR0906` (`dialog_cancellation_rejected`): Dialog cancellation hook rejected/returned veto (runtime-only).
- `AUR0907` (`dialog_cancelled_with_cancel_on_rejection_setting`): `cancelOnReject` option present and dialog cancelled because previous step rejected (runtime-only).
- `AUR0908` (`dialog_custom_error`): Consumer-provided custom error bubbled during dialog lifecycle (runtime-only, content not analyzable statically).
- `AUR0909` (`dialog_closed_before_deactivation`): `dialog.close()` was called before deactivation completed (runtime sequencing issue).

### Notes on Static Analysis
- Most parser/compiler/registration errors can be detected statically by analyzing template/AST structure and resource registrations.
- Lifecycle-dependent codes (e.g., dialog activation rejections, router serialization failures, effect recursion, controller unexpected runtime states, serialization of browser navigation state) depend on runtime data or user-supplied hooks; flag them as potentially thrown but not statically provable.
- Placeholder router codes (`router_error_3`-`router_error_10`) have no defined semantics and should be treated as unknown/unanalysable.
